# Core idea — Forget FastAPI for a moment
![REST API Request Lifecycle](../images/rest_api_request_lifecycle.png)
## Request lifecycle (core truth)

When a client hits an API, the flow looks like:

### **HTTP Request**
- Method (GET / POST / …)
- URL
- Headers
- Body (maybe)

### **Router**
- Matches URL + method
- Calls a Python function

### **Dependencies**
- DB session
- Auth
- Services

### **Business Logic**
- Rules, decisions, orchestration

### **Data Access**
- DB queries
- External APIs

### **HTTP Response**
- Status code
- JSON body

> FastAPI just formalizes this flow.


## Key mental model (remember this)

- **Routes handle HTTP**
- **Services handle logic**
- **Repositories handle data**

If I violate this → unmaintainable system.



# Zooming out (how modern apps work)

Most modern apps work like this: you’ve got a **client**, which is usually your frontend—like a website, a mobile app, or even another service—where the user is actually clicking and interacting.

Then you’ve got the **server**, which is the backend that handles the logic, talks to the database, and sends back the results.



# How they talk: REST

The client and server need a standard way to talk to each other over the internet — that’s where a **REST API** usually comes in.

REST isn’t just “a bridge,” it’s more like a set of rules or a design style for how you expose your backend features in a clean, consistent way over HTTP.

### Resources

In REST, we think in terms of resources — like:

- `books`
- `users`
- `reviews`

Each resource lives at a URL:

- `/books`
- `/users/42`

The client calls these URLs, the server responds with data (usually JSON), and also returns status codes like:

- `200` for “ok”
- `404` for “not found”
- `201` for “created”



# Mapping actions: HTTP methods → CRUD

We interact with those resources using HTTP methods:

| Method | Meaning | Example |
|---|---|---|
| **GET** | read data | `GET /books/123` |
| **POST** | create something new | `POST /books` |
| **PUT** | replace/update a whole resource | `PUT /books/123` |
| **DELETE** | remove something | `DELETE /books/123` |

One small upgrade: a lot of real APIs also use **PATCH**, which means “update part of a resource,” like changing only a rating without sending the whole object again.

# Important REST property: Stateless

REST APIs are usually **stateless**, meaning every request should include everything the server needs — like an auth token — so the server doesn’t rely on remembering your “session” between calls.






# Q&A

### **Question 1:**
If I put database queries directly inside a FastAPI route function, 👉 which architectural rule am I breaking, and why is that dangerous in a real system?

### **Answer 1:**
What’s being violated here is separation of concerns and the layered architecture rule.

In FastAPI, route functions should only deal with HTTP-related stuff — things like the HTTP verb, request/response schemas, status codes, and then calling the service layer. That’s it. Once I start writing database queries inside the route, I’m making the API layer do more than it’s supposed to do.

The service layer should be the one handling the business logic and orchestration, and the repository should be the only place that knows how to talk to the database. If I skip those layers and query the DB directly from the route, I’m breaking that contract.

This is dangerous because it creates tight coupling. Now my HTTP layer depends directly on my database logic. If I ever want to:

- change the database
- reuse the logic somewhere else (background jobs, scripts, tests)
- test things properly without hitting the DB
- or even just refactor safely everything becomes harder.

So instead of having clear, replaceable layers, I end up with a messy API where HTTP, business logic, and data access are mixed together, which doesn’t scale well and becomes painful to maintain in real systems.

> Routes translate HTTP → Services decide → Repositories fetch data
