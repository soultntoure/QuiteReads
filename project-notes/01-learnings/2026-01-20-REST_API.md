So basically, most modern apps work like this: you’ve got a client, which is usually your frontend—like a website, a mobile app, or even another service—where the user is actually clicking and interacting. Then you’ve got the server, which is the backend that handles the logic, talks to the database, and sends back the results.

Now, the client and server need a standard way to talk to each other over the internet, and that’s where a REST API usually comes in. REST isn’t just “a bridge,” it’s more like a set of rules or a design style for how you expose your backend features in a clean, consistent way over HTTP.

In REST, we usually think in terms of resources—like “books,” “users,” “reviews”—and each resource lives at a URL, like /books or /users/42. The client calls these URLs, the server responds with data (usually JSON), and it also returns status codes like 
200
200 for “ok,” 
404
404 for “not found,” or 
201
201 for “created.”

To interact with those resources, we use HTTP methods—these are the actions you mentioned. The common ones map pretty nicely to CRUD:

GET: read data (like GET /books/123)

POST: create something new (like POST /books)

PUT: replace/update a whole resource (like PUT /books/123)

DELETE: remove something (like DELETE /books/123)

One small upgrade: a lot of real APIs also use PATCH, which means “update part of a resource,” like changing only a rating without sending the whole object again. Also, REST APIs are usually stateless, meaning every request should include everything the server needs—like an auth token—so the server doesn’t rely on remembering your “session” between calls.
